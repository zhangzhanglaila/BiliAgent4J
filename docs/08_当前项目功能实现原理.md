# 当前项目功能实现原理

## 1. 总体思路

Java 版实现不是简单把 Python 语法翻译一遍，而是保留原项目的功能边界和接口契约，再按 Spring Boot 分层重组：

- Controller 接口层
- Service 业务层
- Repository 持久化层
- Static 前端资源层

## 2. 内容创作链路

主入口：

```text
/api/module-create
```

大致流程：

1. 整理 `field / direction / idea`
2. 构造 `seed_topic`
3. 抓取热点、分区样本、同类 UP 样本
4. 生成 3 个选题
5. 按选题生成文案

涉及核心类：

- `WorkspaceService`
- `TopicService`
- `TopicDataService`
- `CopywritingService`

## 3. 视频分析链路

主入口：

```text
/api/module-analyze
```

大致流程：

1. 解析视频链接
2. 拿到真实视频信息
3. 判断表现更接近爆款还是低表现
4. 生成优化建议
5. 生成后续选题
6. 筛参考视频
7. 低表现时补一版新文案

涉及核心类：

- `VideoResolverService`
- `OptimizationService`
- `ReferenceVideoService`
- `WorkspaceService`

## 4. 对话链路

主入口：

```text
/api/chat
```

当前 Java 版逻辑：

- 无 Key 时关闭
- 有 Key 时通过 `LlmClientService` 调模型
- 结合当前页面上下文输出回答和下一步建议

## 5. SQLite 在哪里用

`OptimizationService` 会把视频指标历史写入 SQLite，后续可以继续扩展成更完整的趋势分析。

对应类：

- `VideoMetricRepository`

## 6. 为什么拆成多个小 service

因为原 Flask `web/app.py` 既处理路由又处理业务，迁到 Java 后如果还塞进一个大类，后续维护会很差。

所以现在拆成：

- `BilibiliHttpSupport`
- `VideoResolverService`
- `TopicDataService`
- `TopicService`
- `CopywritingService`
- `OptimizationService`
- `ReferenceVideoService`
- `WorkspaceService`

## 7. 当前实现策略

### 规则链路

尽量保持原项目行为一致。

### LLM 链路

先把 LangChain4j 调用链和 fallback 链补齐，保证新项目能独立运行。

### 前端

先沿用现有前端静态资源，保证使用体验连续。

## 8. 按运行模式看完整工作链路

这一节单独按“规则模式”和“LLM Agent 模式”拆开，是为了把同一个功能在当前 Java 项目里到底走了哪条调用链、落到了哪个类的哪个方法，讲得更直观。

### 8.1 启动与运行模式判断

项目启动入口在：

- `com.agent4j.bilibili.Agent4jApplication.main()`
- `com.agent4j.bilibili.config.DotenvLoader.load()`

也就是说，Spring Boot 启动前会先尝试读取 `D:\BiliAgent4J\.env`，把 `LLM_API_KEY`、`LLM_BASE_URL`、`LLM_MODEL` 等配置写入系统属性，作为默认可复用配置来源之一。

运行模式判断在：

- `com.agent4j.bilibili.service.RuntimeLlmConfigService.runtimeLlmEnabled()`
- `com.agent4j.bilibili.service.RuntimeLlmConfigService.runtimeMode()`
- `com.agent4j.bilibili.service.RuntimeInfoService.buildRuntimePayload()`
- `com.agent4j.bilibili.controller.ApiController.runtimeInfo()`
- `com.agent4j.bilibili.controller.ApiController.runtimeMode()`
- `com.agent4j.bilibili.controller.ApiController.runtimeLlmConfig()`

前端会通过 `GET /api/runtime-info` 拿到：

- 当前模式是 `rules` 还是 `llm_agent`
- 聊天助手是否可用
- 页面顶部和助手面板该显示什么说明文案

因此，当前项目的模式切换不是前端自己判断，而是后端统一下发。

当前判断逻辑分成两层：

- `.env` 可以在服务启动时提供一组默认可复用的 LLM 配置
- 页面顶部开关和运行时配置表单决定“当前是否真正启用 LLM Agent”

### 8.2 规则模式链路

规则模式的根判断仍然是：

- `RuntimeLlmConfigService.runtimeLlmEnabled() == false`
- `RuntimeLlmConfigService.runtimeMode() == "rules"`

这时项目仍然会走相同的 API，但业务生成和分析主要依赖本地规则、启发式逻辑和 fallback 结果，不依赖真正的 LLM Agent 中枢。

#### 8.2.1 内容创作模块

前端链路：

- `src/main/resources/static/app.js` 的 `runCreatorModule()`
- 请求 `POST /api/module-create`

后端链路：

- `ApiController.moduleCreate()`
- `WorkspaceService.moduleCreate()`
- `WorkspaceService.buildSeedTopic()` 先把 `field / direction / idea` 整理成种子主题
- `WorkspaceService.refineCreatorProfile()` 生成更自然的创作者画像描述
- `WorkspaceService.runTopic()`
- `TopicService.run()`
- `TopicService.run()` 内部继续调用：
  - `TopicDataService.fetchHotVideos()`
  - `TopicDataService.fetchPartitionVideos()`
  - `TopicDataService.fetchPeerUpVideos()`
  - `TopicService.generateSeedTopics()`
  - `TopicService.generateTrendingTopics()`
  - `TopicService.mergeIdeas()`
- 回到 `WorkspaceService.extractTopTopic()` 选出当前最优选题
- `WorkspaceService.runCopy()`
- `CopywritingService.run()`
- `CopywritingService.run()` 内部会先准备 fallback，再调用：
  - `CopywritingService.fallback()`
  - `CopywritingService.buildTitles()`
  - `CopywritingService.buildScript()`
  - `CopywritingService.buildDescription()`
  - `CopywritingService.buildTags()`
  - `CopywritingService.buildPinnedComment()`
  - `LlmClientService.invokeJson(...)`

这里要注意一层当前 Java 版特有的实现：

- 规则模式下虽然 `CopywritingService` 仍然统一走 `LlmClientService.invokeJson(...)`
- 但 `LlmClientService.available()` 会返回 `false`
- 所以它会直接返回 fallback JSON，而不是去调用远端模型

最终由 `WorkspaceService.moduleCreate()` 返回：

- `seed_topic`
- `normalized_profile`
- `partition`
- `style`
- `topic_result`
- `copy_result`
- `chosen_topic`

这条规则链路的本质是：

- 先由 `TopicService` 做热点样本抓取和启发式选题
- 再由 `CopywritingService` 用本地模板化逻辑生成结构化文案
- 整个过程走的是 Java 本地规则，不消耗 token

#### 8.2.2 视频分析模块

前端链路：

- `app.js` 的 `scheduleVideoResolve()` / `resolveVideoLink()` 先请求 `POST /api/resolve-bili-link`
- `app.js` 的 `runAnalyzeModule()` 再请求 `POST /api/module-analyze`

视频预解析链路：

- `ApiController.resolveBiliLink()`
- `WorkspaceService.resolveBiliLink()`
- `VideoResolverService.resolveVideoPayload()`
- `VideoResolverService.extractBvid()`
- `BilibiliHttpSupport.resolveShortLink()` 负责展开短链
- `VideoResolverService.fetchVideoInfo()`
- `VideoResolverService.fetchVideoInfoViaPublicApi()`
- 如果公开视频接口失败，再回退到：
  - `VideoResolverService.fetchVideoInfoViaHtml()`
  - `VideoResolverService.normalizeHtmlInfo()`
- 最后由 `VideoResolverService.buildResolvedPayload()` 统一产出 `resolved`

规则分析链路：

- `ApiController.moduleAnalyze()`
- `WorkspaceService.moduleAnalyze()`
- 如果前端带来的 `resolved` 仍然可复用，会先走：
  - `VideoResolverService.isResolvedPayloadUsable()`
- 如果不能复用，则重新走：
  - `VideoResolverService.resolveVideoPayload()`
- 然后构造市场快照：
  - `WorkspaceService.buildMarketSnapshot()`
  - `TopicDataService.fetchHotVideos()`
  - `TopicDataService.fetchPartitionVideos()`
  - `TopicDataService.fetchPeerUpVideos()`
- 再进入规则分析主链：
  - `WorkspaceService.runTopic()`
  - `TopicService.run()`
  - `OptimizationService.run()`
  - `WorkspaceService.classifyVideoPerformance()`

其中 `OptimizationService.run()` 内部继续调用：

- `OptimizationService.fetchVideoMetrics()`
- `VideoResolverService.fetchVideoInfo()`
- `OptimizationService.estimateCompletionRate()`
- `OptimizationService.ruleBasedDiagnosis()`
- `VideoMetricRepository.saveVideoMetrics()`
- `LlmClientService.invokeJson(...)`

同样地，规则模式下这里的 `invokeJson(...)` 也会直接落回 fallback 结果，所以最终仍然是本地规则分析。

后续分支为：

- 如果判断为热门视频：
  - `WorkspaceService.buildHotAnalysis()`
- 如果判断为低表现视频：
  - `WorkspaceService.runCopy()`
  - `CopywritingService.run()`
  - `WorkspaceService.buildLowPerformanceAnalysis()`

参考视频部分由：

- `ReferenceVideoService.buildReferenceQueryText()`
- `ReferenceVideoService.buildReferenceVideosFromMarketSnapshot()`
- `ReferenceVideoService.selectReferenceVideos()`
- `ReferenceVideoService.enrichReferenceSourcesWithSearch()`
- `ReferenceVideoService.fetchDirectRelatedReferenceVideos()`
- `ReferenceVideoService.fetchSearchReferenceVideos()`
- `ReferenceVideoService.buildRank()`

这一条规则链路的本质是：

- 先把 B 站公开视频信息解析成统一结构
- 再用 Java 本地阈值和启发式规则判断“更像热门”还是“更像低表现”
- 最后补充选题、优化建议、参考视频和必要时的新文案

#### 8.2.3 智能对话助手

规则模式下不会真正启用聊天能力，对应控制点在：

- `RuntimeInfoService.buildRuntimePayload()`
- `ApiController.runtimeInfo()`
- `WorkspaceService.chat()`
- `RuntimeLlmConfigService.runtimeLlmEnabled()`

具体表现是：

- 前端通过 `/api/runtime-info` 拿到 `chat_available=false`
- `app.js` 会把助手输入框和发送按钮设为 `disabled`
- 如果仍然直接请求 `POST /api/chat`，`WorkspaceService.chat()` 会抛异常

也就是说，规则模式下聊天面板仍然保留在页面里，但它只是一个关闭状态的 UI，不会进入后端问答链路。

### 8.3 LLM Agent 模式链路

LLM Agent 模式下，入口判断仍然是：

- `RuntimeLlmConfigService.runtimeLlmEnabled() == true`
- `RuntimeLlmConfigService.runtimeMode() == "llm_agent"`

区别在于：`WorkspaceService` 的几个核心入口会优先切到 `LlmWorkspaceAgentService`，由 LangChain4j 驱动的 Agent 中枢先做决策、工具调用和最终 JSON 生成。

#### 8.3.1 Agent 中枢怎么工作

当前 Java 版的 Agent 中枢在：

- `LlmWorkspaceAgentService.runStructured()`

它不是直接把提示词发给模型，而是显式做了这些约束：

- 只允许调用白名单工具
- 可要求某些工具必须先被调用
- 可要求最终 JSON 必须包含指定字段
- 记录 `agent_trace`
- 记录 `tool_observations`
- 用 `maxSteps` 限制工具调用轮数

工具注册入口在：

- `LlmWorkspaceAgentService.registeredTools()`

当前注册的工具有：

- `creator_briefing`
- `video_briefing`
- `hot_board_snapshot`

这些工具的实际实现分别是：

- `LlmWorkspaceAgentService.buildCreatorBriefing()`
- `LlmWorkspaceAgentService.buildVideoBriefing()`
- `LlmWorkspaceAgentService.buildHotBoardSnapshot()`

#### 8.3.2 内容创作模块

前端链路不变：

- `app.js` 的 `runCreatorModule()`
- 请求 `POST /api/module-create`

后端链路改成：

- `ApiController.moduleCreate()`
- `WorkspaceService.moduleCreate()`
- `LlmWorkspaceAgentService.runModuleCreate()`
- `LlmWorkspaceAgentService.runStructured()`

内容创作模式下允许且要求调用的工具是：

- `creator_briefing`

工具实际落点是：

- `LlmWorkspaceAgentService.buildCreatorBriefing()`
- `buildCreatorBriefing()` 内部会继续调用：
  - `AppProperties.normalizePartition()`
  - `LlmWorkspaceAgentService.buildMarketSnapshot()`
  - `TopicDataService.fetchHotVideos()`
  - `TopicDataService.fetchPartitionVideos()`
  - `TopicDataService.fetchPeerUpVideos()`

因此，LLM 在内容创作模式下拿到的不是一段裸文本，而是：

- 用户原始输入
- 规范化后的分区
- 热榜样本
- 分区样本
- 同类 UP 样本

Agent 产出最终 JSON 后，`WorkspaceService.moduleCreate()` 直接把结果返回前端。

如果 Agent 中枢失败，则会回退到：

- `WorkspaceService.runLlmModuleCreateFallback()`
- `LlmClientService.invokeJsonRequired()`

这个 fallback 仍然会做本地兜底：

- `WorkspaceService.refineCreatorProfile()`
- `WorkspaceService.buildSeedTopic()`
- `WorkspaceService.runTopic()`
- `CopywritingService.run()`

所以 LLM 模式下的内容创作本质上是：

- 先通过 `creator_briefing` 把市场样本准备好
- 再由 `LlmWorkspaceAgentService` 按约束产出结构化 JSON
- 如果 Agent 失败，再切到单次 LLM JSON 兜底

#### 8.3.3 视频分析模块

前端链路仍然一样：

- `resolveVideoLink()` 先拿 `resolved`
- `runAnalyzeModule()` 再发 `POST /api/module-analyze`

后端链路改成：

- `ApiController.moduleAnalyze()`
- `WorkspaceService.moduleAnalyze()`
- 如果需要，仍然先走：
  - `VideoResolverService.isResolvedPayloadUsable()`
  - `VideoResolverService.resolveVideoPayload()`
  - `VideoResolverService.extractBvid()`
  - `VideoResolverService.fetchVideoInfo()`
  - `VideoResolverService.buildResolvedPayload()`
- 然后后端先构造市场快照：
  - `WorkspaceService.buildMarketSnapshot()`
- 再进入：
  - `LlmWorkspaceAgentService.runModuleAnalyze()`
  - `LlmWorkspaceAgentService.runStructured()`

视频分析模式下允许使用的工具是：

- `hot_board_snapshot`

工具实际落点是：

- `LlmWorkspaceAgentService.buildHotBoardSnapshot()`
- `buildHotBoardSnapshot()` 内部继续调用 `LlmWorkspaceAgentService.buildMarketSnapshot()`

也就是说，当前 Java 版的视频分析 Agent 并不是自己去解析视频真实数据，而是：

- 真实视频信息由 `VideoResolverService` 先整理好
- 热榜和分区样本由后端先准备好
- LLM 主要基于这些结构化输入做判断、分析和建议输出

如果 Agent 中枢失败，还会回退到：

- `WorkspaceService.runLlmModuleAnalyzeFallback()`
- `LlmClientService.invokeJsonRequired()`

fallback 返回后，后端还会继续补齐：

- `WorkspaceService.classifyVideoPerformance()`
- `WorkspaceService.runTopic()`
- `OptimizationService.run()`
- `WorkspaceService.buildHotAnalysis()` 或 `WorkspaceService.buildLowPerformanceAnalysis()`
- `ReferenceVideoService.buildReferenceVideosFromMarketSnapshot()`

所以 LLM 模式下的视频分析，本质上是：

- 视频真实信息仍由后端先解析
- 市场样本仍由后端先准备
- Agent 成功时由模型输出结构化分析
- Agent 失败时由 direct LLM fallback 加本地规则兜底

#### 8.3.4 智能对话助手

这个功能只在 LLM Agent 模式下真正启用。

前端链路：

- `app.js` 的 `sendAssistantMessage()`
- 请求 `POST /api/chat`

后端链路：

- `ApiController.chat()`
- `WorkspaceService.chat()`
- 优先进入：
  - `LlmWorkspaceAgentService.runChat()`
  - `LlmWorkspaceAgentService.runStructured()`

聊天模式下允许的工具是：

- `creator_briefing`
- `video_briefing`
- `hot_board_snapshot`

这些工具对应的实现分别是：

- `LlmWorkspaceAgentService.buildCreatorBriefing()`
- `LlmWorkspaceAgentService.buildVideoBriefing()`
- `LlmWorkspaceAgentService.buildHotBoardSnapshot()`

其中：

- `buildVideoBriefing()` 会继续调用：
  - `VideoResolverService.extractBvid()`
  - `VideoResolverService.fetchVideoInfo()`
  - `VideoResolverService.buildResolvedPayload()`
  - `LlmWorkspaceAgentService.buildMarketSnapshot()`
- `buildCreatorBriefing()` 会继续调用：
  - `AppProperties.normalizePartition()`
  - `LlmWorkspaceAgentService.buildMarketSnapshot()`

Agent 正常完成时，当前 Java 版聊天接口至少返回：

- `reply`
- `suggested_next_actions`
- `mode`

同时可能附带：

- `agent_trace`
- `tool_observations`

如果 Agent 中枢失败，则 `WorkspaceService.chat()` 会回退到：

- `LlmClientService.invokeText()`

并在有视频链接时额外调用：

- `VideoResolverService.resolveVideoPayload()`
- `WorkspaceService.buildMarketSnapshot()`
- `ReferenceVideoService.buildReferenceQueryText()`
- `ReferenceVideoService.buildReferenceVideosFromMarketSnapshot()`

因此，当前 Java 版聊天助手的本质不是单纯问答，而是：

- 先看用户有没有给创作上下文
- 再看有没有给视频链接
- 需要时动态调用市场样本工具和视频解析工具
- 最后返回回答和下一步建议
